# Dev分支双轨里程碑（4周）设计

## 1. 背景与现状
`dev` 分支已完成数据任务中心的第一阶段落地，包含任务注册表、状态模型、API、Celery任务、基础UI与测试基线。

已落地能力：
- 数据任务模型与迁移：`app/models/data_job_run.py`、`app/models/data_job_cursor.py`、`migrations/versions/20260301_add_data_job_tables.sql`
- 服务与执行链路：`app/services/data_jobs/*`、`app/tasks/data_jobs_tasks.py`、`app/celery_app.py`
- API与页面入口：`app/api/data_jobs_api.py`、`app/templates/realtime_analysis/index.html`、`app/static/js/data_jobs.js`
- 相关测试：`tests/data_jobs/*`、`tests/api/test_data_jobs_api.py`、`tests/ui/test_data_jobs_ui_contract.py`

仍需补齐的核心缺口：
- 数据任务中心偏MVP：缺取消、去重互斥、细粒度进度与运维视图
- 策略主线关键能力仍有占位实现：回测基准、自定义因子、集成评分、高级优化
- 测试覆盖偏单点：集中在 data_jobs，策略主链路缺系统化回归

## 2. 目标与验收
在 4 周内并行推进两条轨道，并在每周形成可验证交付：

- 工程轨：把数据任务中心从“可用”提升为“可运营”
- 策略轨：把因子-评分-优化-回测主链路中的关键占位能力逐步替换为可用实现

统一验收口径：
1. 每周至少一个后端能力闭环（API + service + test）
2. 每周至少一个策略能力闭环（实现 + 回测/评估验证）
3. 每周有可复现结果（参数、时间窗口、输出摘要）

## 3. 方案比较与决策
### 方案A：仅工程轨
- 优点：系统稳定性提升快
- 缺点：策略能力推进慢，业务价值滞后

### 方案B：仅策略轨
- 优点：策略指标可能更快改善
- 缺点：运行稳定性、可观测性不足，返工风险高

### 方案C：双轨里程碑（选定）
- 每周并行交付工程与策略各一组能力
- 通过周验收降低“只做一边”的结构性风险

## 4. 目标架构（增量）
### 4.1 工程轨
- `DataJobService` 负责任务编排与策略（提交、重试、取消、去重）
- `DataJobStateStore` 负责状态持久化（运行态、错误、游标、统计）
- `data_jobs_api` 负责统一接口契约（清晰错误码、分页查询、状态过滤）
- `data_jobs.js` 承担前端交互（提交、轮询、日志、历史）

### 4.2 策略轨
- `BacktestEngine` 增加基准收益链路，形成策略/基准对比
- `FactorEngine` 增加安全表达式因子能力
- `StockScoringEngine` 增加ML集成评分与RankIC权重
- `PortfolioOptimizer` 增加因子中性约束（BL延后）

## 5. 数据流与执行流
### 5.1 数据任务中心
1. 前端提交任务参数到 `POST /api/data-jobs/submit`
2. service层校验任务类型与运行策略（去重/互斥）
3. 创建 run 记录并入队
4. worker 执行脚本并持续写入状态
5. 前端轮询 run 详情并展示进度/日志

### 5.2 策略主链路
1. 因子计算（内置 + 自定义）
2. 评分融合（因子 + 模型）
3. 组合构建（约束优化）
4. 回测执行（含基准对比）
5. 报告输出（收益、风险、稳定性）

## 6. 错误处理与可观测性
- API 层区分 4xx/5xx，避免业务异常统一返回 500
- 任务运行输出结构化结果（return code、stdout/stderr截断、错误类型）
- 状态流转明确：`pending -> queued -> running -> success|failed|cancelled`
- 为长任务补充取消信号和超时策略

## 7. 测试策略
- 工程轨：API契约测试、状态机测试、任务调度/重试测试、UI契约测试
- 策略轨：回测指标单测、因子表达式单测、评分融合单测、优化约束单测
- 每周回归：至少一次 `pytest` 全量关键模块回归

## 8. 4周里程碑
### 第1周
- 工程：任务历史/详情/日志查询闭环 + 错误码分级
- 策略：回测基准（HS300）接入与对比输出

### 第2周
- 工程：任务互斥/去重 + 失败重试策略
- 策略：自定义因子表达式MVP（白名单）

### 第3周
- 工程：任务取消 + 超时治理 + 统计看板
- 策略：ML集成评分 + RankIC动态权重评分

### 第4周
- 工程：关键链路测试补齐 + 最小CI门禁
- 策略：因子中性优化落地（BL顺延到下一阶段）

## 9. 风险与缓解
1. 脚本侧能力不一致导致任务编排复杂
- 缓解：先统一 runner 参数契约，再做脚本深改

2. 策略增强影响现有回测结果稳定性
- 缓解：保留旧逻辑开关，逐步切换并建立回归基线

3. 测试数据依赖数据库状态
- 缓解：优先补充纯单元测试与可控fixture，降低环境耦合

## 10. 本阶段边界
- 不含实盘交易接入
- 不含分钟级全量策略框架重构
- Black-Litterman 仅保留接口与设计，不在本4周内强行落地
