# 数据下载页面化整合设计（realtime-analysis 扩展）

## 1. 背景与目标
当前项目的数据下载能力主要集中在 `app/utils/*.py` 脚本中，执行方式依赖手工命令行触发，缺少页面入口、进度可视化、失败重试与审计追踪，导致运维成本高、可用性低。

本设计目标：
- 将 `app/utils` 全量数据任务整合到页面（入口放在 `realtime-analysis`）。
- 支持异步执行、实时进度、日志查看、历史追踪。
- 默认采用增量更新策略；全量重建作为危险操作需二次确认。
- 按数据域组织任务，降低使用门槛。

## 2. 约束与已确认决策
- 页面入口：扩展 `realtime-analysis`（不是新建 ml-factor 页面）。
- 覆盖范围：`app/utils` 全量任务。
- 执行模型：异步任务队列。
- 数据策略：默认增量更新；全量重建需强确认。
- 页面组织：按数据域分组展示任务。

## 3. 方案对比与选型
### 方案A：脚本包装直跑（最小改造）
- 保留脚本，页面触发 subprocess 执行。
- 优点：上线快。
- 缺点：参数硬编码与幂等问题保留，长期维护差。

### 方案B：分层重构（选定）
- 建立任务服务层，对脚本进行 Runner 适配，统一 API 与任务状态管理。
- 优点：兼顾上线速度与可维护性，便于后续扩展调度/权限/告警。
- 缺点：首期改造量中等。

### 方案C：平台化 DAG 调度中心
- 一次性建设编排平台。
- 优点：能力完整。
- 缺点：周期长、风险高。

结论：选择 **方案B**。

## 4. 总体架构
采用四层架构：
1. 页面层：`realtime-analysis` 新增“日频数据中心”标签页。
2. API 层：`app/api/data_jobs_api.py` 提供提交/查询/取消/重试接口。
3. 任务服务层：`app/services/data_jobs/` 提供 Runner 与 Registry。
4. 执行与持久化层：Worker + 任务状态表（状态、进度、日志、统计）。

## 6. 数据流与执行时序
1. 页面提交任务参数至 `POST /api/data-jobs/submit`。
2. API 校验参数与依赖，创建任务实例并入队。
3. Worker 拉取任务并执行，状态 `pending -> running`。
4. Runner 持续写入进度与日志。
5. 成功写入结果统计并置 `success`。
6. 失败写错误详情并置 `failed`，支持页面重试。
7. 前端轮询查询任务详情与历史。

## 7. 错误处理、幂等与安全策略
- 默认禁止 `truncate`，改为增量模式。
- 全量重建仅在 `full_refresh=true` 且二次确认后允许。
- 写库策略统一为 `INSERT ... ON DUPLICATE KEY UPDATE` 或 `INSERT IGNORE`。
- 可恢复异常自动重试（默认 3 次，指数退避）。
- 保存断点游标（日期/offset），失败后可续跑。
- 同任务类型+同参数集互斥，避免重复并发。
- 记录触发人、参数快照、耗时、结果统计与失败原因。

## 8. 页面交互设计（realtime-analysis）
- 新增“日频数据中心”主区域。
- 左侧任务分组列表，右侧参数表单与说明。
- 执行控制：启动/取消/重试/查看最近结果。
- 运行状态：进度条 + 阶段描述 + 实时日志。
- 历史记录：支持按任务类型、状态、时间筛选。
- 危险操作弹窗：必须输入确认语后才能提交全量任务。

## 9. 测试与验收标准
验收标准：
1. 全量 `app/utils` 任务可从页面触发。
2. 任务支持可视化进度、日志、历史查询。
3. 默认增量更新，未确认时不会触发全量重建。
4. 任务失败可重试且支持断点续跑。
5. 同类同参任务不可重复并发。

测试建议：
- 单元：注册表、参数校验、依赖判定。
- 集成：提交-执行-状态流转-结果落库。
- 失败注入：限流/超时/断连/脏数据。
- UI：进度刷新、日志显示、危险确认流程。

## 10. 资源与外部依赖（开发中需补齐）
1. 队列选型与实例：推荐 `Celery + Redis`。
2. Redis 地址与鉴权配置。
3. 任务状态表迁移方案（Alembic 或 SQL 迁移）。
4. 权限策略（是否管理员专属全量重建）。
5. Tushare/Baostock 频控阈值与默认节流参数。
6. 部署方式（单进程/多进程）与 Worker 启停策略。

## 11. 风险与分阶段建议
- 风险：旧脚本硬编码、表结构不统一、历史数据质量不一致。
- 建议：先做 Runner 适配层确保页面可用，再逐步深度重构高频任务。
